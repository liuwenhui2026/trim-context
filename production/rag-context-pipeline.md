# RAG + 上下文工程完整链路设计

> 🚧 Work in Progress

RAG 不是孤立的——它是上下文工程"Select"环节的一部分。这里讲的是从用户提问到模型回答，上下文怎么流转。

## 完整链路

```
用户提问
  → Query Rewrite（改写查询，提升召回）
  → Retrieval（检索，可能返回 20+ 条）
  → Rerank（重排序，按相关性排序）
  → Trim（截断，只保留 Top-K 或按预算裁剪）     ← 截断在这里
  → Context Assembly（组装：系统指令 + 对话历史 + RAG 结果 + 工具定义）
  → Token Budget Check（检查是否超窗口）
      ├─ 没超 → 直接发送
      └─ 超了 → 压缩对话历史 / 进一步裁剪 RAG 结果 / 折叠工具输出
  → LLM 推理
  → 回答
```

## 关键决策点

1. RAG 结果占上下文多少预算？
2. 对话历史和 RAG 结果冲突时，优先保留谁？
3. 多轮对话中，之前的 RAG 结果要不要保留？
